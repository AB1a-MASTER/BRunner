WORK COMPLETED

1. Recording and replay reliability
- Recorded elements carry semantic target candidates, snapshots, page context,
  and compatibility fields.
- Replay uses direct candidates, the controls tree, and fuzzy document fallback.
- Failed resolution reports useful diagnostics instead of failing on one selector.
- Same-page, cross-page, and multi-tab recording/replay have been tested.
- Restricted pages and the Studio are excluded from content recording.
- Recording on a restricted New Tab no longer borrows the pinned tab's domain.

2. Multi-tab runtime
- Tabs have workflow-local logical tab references (tabRef).
- Recorded tab switches can be replayed or recovered if a child tab is missing.
- Open Tab can continue in the current tab or move execution to the new tab.
- Close Tab returns to an appropriate remaining/opener tab and does not create an
  empty replacement tab.
- Element nodes can target a selected logical tab.

3. Workflow persistence and synchronization
- Saving an edited workflow updates the original instead of creating a copy.
- Renaming from Studio updates the workflow filename/name correctly.
- Duplicate remains a separate explicit operation.
- New recordings and runtime state synchronize between Studio and sidebar.
- Native-host save/load compatibility was retained.

4. Startup and browser behavior
- A workflow's bound domain is honored at startup.
- The current usable tab navigates to that domain by default without replacing
  Studio.
- Reusing an already-open matching tab is optional and off by default.
- A default-browser-search node is available.
- Back, forward, reload, open tab, close tab, and recorded tab switch are present.

5. Shared runtime state and cancellation
- Studio and sidebar share authoritative recording/execution status.
- Workflows can be stopped from Studio, the sidebar main control, or a running
  workflow row.
- Sidebar controls change from Play/Start to Stop while execution is active.
- Cancellation interrupts long background waits and page-level conditional waits,
  prevents later steps, and reports cancelled rather than failed.

6. Node and data foundation
- A canonical node registry drives the Studio palette/configuration.
- Every run receives a fresh variable registry.
- {{variable}} expressions resolve immediately before each step and fail clearly
  when a variable is missing.
- Extraction nodes cover text, attributes, lists, tables, and page metadata.
- Data nodes include set variable and template.
- Wait supports fixed and randomized timing.
- Sequential browser/element/wait/data-transform nodes from Phase A are present:
  double-click, hover, clear input, element/page scrolling, conditional waits,
  JSON parse/stringify, regex match/replace, number conversion, and date format.

7. Documentation and planning

- docs/specs/01_MULTI_TAB_RUNTIME.md defines tab/runtime behavior.
- docs/specs/02_DATA_NODE_ENGINE.md defines variables and extraction.
- docs/specs/03_STUDIO_GRAPH_UX.md defines the later graph editor migration.
- docs/specs/04_NODE_CATALOG.md defines phased node expansion.

8. Phase B external and file operations
- HTTP Request runs in the background and supports GET, POST, PUT, PATCH, DELETE,
  and HEAD; text/JSON bodies and responses; timeout; non-2xx policy; structured
  output; and AbortController workflow cancellation.
- HTTP requests omit browser credentials by default. Request headers, bodies,
  URL query strings, and embedded credentials are excluded from execution logs.
- Clipboard Read and Clipboard Write use a Chrome offscreen document. Reads
  default to denied and require explicit node-level approval. Clipboard values
  are excluded from logs.
- Clipboard operations use navigator.clipboard when available and an
  execCommand copy/paste fallback for offscreen-document focus restrictions.
- Upload Virtual File creates a real File from expression-enabled text or base64
  content, assigns it to a web input[type=file], dispatches input/change events,
  and stores name/type/size metadata in a workflow variable.
- Virtual uploads reject filesystem paths, cap payloads at 10 MB, and exclude
  file content from logs. Arbitrary local paths/native dialogs remain deferred
  until native-access permissions are explicitly designed.
- Toggle true/false behavior for checkbox and radio controls was corrected.
- Deterministic suite currently passes 17 tests. Toggle, HTTP, clipboard, and
  virtual file-input workflows all passed live acceptance.

NEXT IMPLEMENTATION PLAN

Step 1 - Implement Download Wait and metadata
- Design explicit browser download permission and safe filename/path metadata.
- Add a registry-driven node that waits for a matching download with timeout and
  cancellation support.
- Return safe metadata such as id, filename, URL/final URL, MIME type, bytes,
  state, danger classification, and timestamps where Chrome exposes them.
- Never log sensitive download URLs or full local paths unless explicitly needed.
- Distinguish timeout, interruption, danger, cancellation, and missing-permission
  diagnostics.
- Add deterministic tests and one live acceptance workflow.

Step 2 - Implement Screenshot Capture
- Decide browser-tab capture versus page/element capture and output destination.
- Add explicit format/quality/full-page behavior where technically supported.
- Return attachment/path metadata without silently persisting sensitive images.
- Add cancellation, permission review, deterministic tests, and live acceptance.

Step 3 - Design native file access separately
- Add arbitrary local-path/native-dialog upload only through an explicit native
  host permission and approval model.
- Do not broaden virtual file upload into silent filesystem access.

Each external/file node needs a permission and secret-handling review before code.

Step 4 - Studio UX milestone
- Build the Variables/Data Inspector and previews first.
- Add stronger node-specific panels, validation, and variable autocomplete.
- Then migrate to graph schema v2/React Flow with safe v1 upgrades and backups.

Step 5 - Graph-dependent automation
- If/else, switch, merge, stop/fail, and retry boundaries.
- Repeat, while, and for-each with loop limits.
- Sub-workflows and mapped inputs/outputs.
- Error branches and reusable workflow components.

DO NOT IMPLEMENT GRAPH CONTROL FLOW IN THE CURRENT LINEAR EXECUTOR. It depends on
the graph schema defining branch handles, traversal, cycle validation, and limits.
